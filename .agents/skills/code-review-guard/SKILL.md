# 代码交付前自审守卫

本 Skill 在模块开发完成后触发，对照 `.spec.md`、`tasks.yaml` 和 `acceptance.yaml` 验证实现是否完整。

## 使用方式

1. 开发完成后，打开对应模块的 `.spec.md`、`tasks.yaml`、`acceptance.yaml`
2. 逐项对照 .spec 中的功能需求，检查代码是否实现
3. 对照 `tasks.yaml` 检查任务是否真的满足 DoD，而不是只改了代码
4. 使用 `fullstack-dev-checklist` Skill 的九大维度再次全面检查
5. 若准备对外宣称“完成”，必须再调用 `acceptance-gate`
6. 输出验收报告

---

## 自审流程

### Step 0：工程完整性验证 [硬性前置]

**在进行任何逻辑审查之前，先确认代码能编译、能运行、能访问：**

#### A. 编译与入口
- [ ] 执行过项目级编译/构建命令且无错误？
- [ ] 新模块的接口处理器已注册到路由文件？
- [ ] 新模块的服务链路已在主入口文件中完成初始化？
- [ ] 新增的 Model 已加入数据库迁移脚本？
- [ ] 同一目录下无重复定义的结构体/类/函数？
- [ ] 增量开发场景下，新模块引用的旧 Model 已补全所有必要字段？

#### B. 项目脚手架与可运行性
- [ ] 前端项目骨架是否齐全（依赖声明 + 构建配置 + HTML入口 + 应用初始化 + 根组件/布局 + 路由表）？
- [ ] 后端项目骨架是否齐全（依赖声明 + 启动入口 + 路由注册 + 中间件）？
- [ ] 是否有容器化构建文件和环境编排文件能一键启动完整环境？
- [ ] 后端是否支持开发模式鉴权旁路？
- [ ] 前后端 API 代理配置是否匹配？HTTP 工具是否已封装错误处理/Token 挂载？
- [ ] 是否存在版本控制忽略规则文件？

**未通过 Step 0 的代码，直接判为「不合格」，不进入后续审查。**

### Step 1：.spec 对照检查

逐一检查 .spec 文件中定义的每个功能需求（FR-XXX）：

```
对于 .spec 中的每个 FR-XXX：
  1. 该功能是否已实现？
  2. 交互流程是否与 .spec 描述一致？
  3. 数据字段是否完整（.spec 字段表中的每一行）？
  4. 字段的输入组件类型是否合理？
  5. 状态处理（加载/空/错误/成功）是否实现？
  6. 权限检查是否到位？
  7. 边界条件是否处理？
```

### Step 2：九维度全面检查

引用 `fullstack-dev-checklist` Skill，逐维度检查：

0. **工程完整性**：编译通过、入口注册、增量兼容（已在 Step 0 完成）
1. **页面完整性**：列表页/表单页/详情页的所有要素
2. **API 完整性**：参数校验、错误码、关联数据预加载
3. **数据库设计**：索引、外键、字段类型
4. **用户体验**：加载状态、操作反馈、表单体验
5. **前后端一致性**：字段名、类型、校验规则
6. **业务逻辑合理性**：关联数据展示、状态流转、级联影响
7. **无障碍与核心体验**：A11y、语义化 HTML、Web Vitals
8. **双向溯源**：代码注释 ↔ .spec 锚点映射

### Step 3：深度审计与专项检查

完成功能验证后，根据被 review 的代码模块属性，决定是否**强制调用**以下连带专项审查 Skill：

1. **如涉及订单、金额、权限流转** 👉 立即启动 `security-guard` 进行防越权与防并发覆盖审计。
2. **如涉及复杂架构设计、大范围重构** 👉 立即启动 `architecture-reviewer` 进行 SOLID 原则与内聚性审计。
3. **如为常规CRUD页面** 👉 继续检查以下高频遗漏项：

#### 页面层面
按 `fullstack-dev-checklist` **维度一（页面完整性）+ 维度四（UX）** 重点复查以下高频遗漏：
- [ ] 关联数据选择器是否显示名称（而非 ID）
- [ ] 危险操作是否有二次确认弹窗（含对象名称）
- [ ] 表单提交成功后是否跳转回列表并刷新，是否做了防重复提交处理
- [ ] 数据格式化是否到位（时间可读、金额带单位、状态用标签、下拉有占位符）
- [ ] 文件上传全链路是否到位（前端组件 + 后端接口 + URL 返回）

#### API 层面
- [ ] 列表接口是否返回关联数据的名称
- [ ] 创建/更新接口是否有完整的参数校验
- [ ] 分页参数是否正确处理
- [ ] 错误响应是否使用统一格式
- [ ] **实际调用烟雾测试**：每个接口是否实际调用过一次（包括创建/查询/更新/删除/状态变更），并**附上实际请求和响应日志作为证据**？

#### 回归影响
- [ ] 本次修改是否涉及共享代码（Model、工具函数、中间件、共享组件）？
- [ ] 若涉及共享代码，关联模块是否已重新编译/测试确认无影响？
- [ ] 修复 Bug 时，是否确认修复不会引入新的问题？

#### 数据层面
- [ ] 外键字段是否有索引
- [ ] 金额是否用定点数类型
- [ ] 软删除是否正确实现
- [ ] 事务是否在需要的地方使用

### Step 4：交付证据与验收门禁

- [ ] 是否存在构建结果、测试结果、关键日志、截图、接口样例等 Evidence Pack？
- [ ] `tasks.yaml` 中的关键任务是否都附有完成证据？
- [ ] 若存在 blocker，是否明确记录而不是模糊写成“待优化”？
- [ ] 是否调用 `acceptance-gate` 给出最终通过 / 阻塞 / 建议优化结论？

## 验收报告模板

```markdown
# 验收报告：[模块名]

## 基本信息
- **模块名称**：[名称]
- **.spec 文件**：[路径]
- **审查日期**：[日期]
- **总体评分**：[X/10]

## .spec 对照结果

| FR编号 | 功能描述 | 实现状态 | 备注 |
|--------|---------|---------|------|
| FR-001 | [描述] | ✅ 完整 | |
| FR-002 | [描述] | ⚠️ 部分 | [缺少什么] |
| FR-003 | [描述] | ❌ 未实现 | [原因] |

## 九维度检查结果

| 维度 | 完成度 | 关键发现 |
|------|--------|---------|
| 工程完整性 | 100% | 编译通过、入口齐全 |
| 页面完整性 | 90% | 缺少空状态处理 |
| API 完整性 | 85% | 缺少关联数据预加载 |
| 数据库设计 | 100% | 无问题 |
| 用户体验 | 70% | 缺少操作反馈 |
| 前后端一致性 | 95% | 分页参数起始值不一致 |
| 业务逻辑合理性 | 80% | 关联数据显示ID而非名称 |
| 无障碍/体验 | 60% | 缺少语义化标签 |
| 双向溯源 | 75% | 部分核心逻辑缺少 .spec 注释 |

## 遗漏项清单

### 必须修复（阻塞交付）
1. [ ] [具体描述]
2. [ ] [具体描述]

### 建议优化（不阻塞但应改进）
1. [ ] [具体描述]

## 结论
- [ ] ✅ 通过验收，可交付
- [ ] ⚠️ 需修复上述必须项后重新验收
- [ ] ❌ 多处关键功能缺失，需要重构
```

## 触发时机

本 Skill 在以下情况被触发：
1. 开发任务标记为"完成"前
2. 用户要求"检查一下代码是否完整"
3. 提交 PR / Code Review 前
4. 用户说"这个模块做完了"

触发后，AI 必须：
1. 自动定位对应的 .spec 文件
2. 执行完整的自审流程和交付证据检查
3. 输出格式化的验收报告
4. 如有未通过项，列出具体待修补清单
