# AI-OS 项目目的

## 1. 最终目的

AI-OS 的最终目的，是让 AI 在不同工具里都能以更稳定、更可恢复、更可交付的方式参与项目，而不是只产出零散代码或一次性回答。

它服务的是已经开始用 AI 做开发，但流程还不稳定、容易丢上下文、容易越做越偏、也不希望为每个工具各维护一套规则的个人开发者和小团队。

因此，AI-OS 的核心工作不是堆更多条目，而是用尽量少、但足够稳定的要求，把 AI 的工作方式约束到真正有助于交付的方向上。

## 2. 六条核心要求

### 2.1 目标对齐

AI-OS 只优化 AI 驱动的项目交付质量、稳定性、可恢复性和新手可理解性。

任何新增要求，如果不能直接提升这些目标之一，就不应进入 AI-OS 的核心层。

### 2.2 通用性优先

只有跨项目复现、可抽象的问题才进入框架。

行业偏好、团队习惯、单个项目的特殊做法，不进入 AI-OS 的核心要求。

### 2.3 项目事实边界

AI-OS 负责通用交付机制，业务边界、阶段目标、项目偏好和具体上下文放在项目工件中。

任何输出默认服务当前项目，而不是参考系统、历史系统或 AI 自带的默认习惯。

### 2.4 闭环与证据优先

AI-OS 优先保证可运行闭环、依赖完整性和证据化完成，而不是局部页面完成、模块列表完成或口头“已完成”。

### 2.5 复杂度预算

AI-OS 的核心概念、workflow、工件和默认行为必须保持受控。

新增能力默认应复用、替换或合并旧结构，而不是继续叠加。

### 2.6 结构化演进

核心原则必须有主定义位置，框架级变更要有证据，并同步清理被替代的旧规则、旧文案和旧测试。

AI-OS 的演进方式应是结构化重编排，而不是“用户提一个点就补一条”。

## 3. 新需求筛选法

每个关于 AI-OS 本身的新需求，都先回答这 4 个问题：

1. 这是 AI-OS 的通用交付问题，还是某个项目、行业、团队的使用问题？
2. 它是否直接提升六条核心要求中的至少一条？
3. 它最合适的承接位置是 `framework/`、项目工件、README/示例，还是明确不纳入？
4. 如果进入 `framework/`，应新增，还是应合并、重写、抽象提升或替换现有结构？

允许的结论只有 4 类：

- 进入 `framework/`
- 留在项目工件
- 通过 README / 示例 / 使用方式解决
- 明确不纳入 AI-OS

如果一个需求不能稳定映射到六条核心要求之一，默认不进入根层治理文档。

## 4. 派生规则原则

具体规则可以存在，但不应再与核心要求并列。它们都应视为从六条核心要求推导出来的派生规则。

派生规则的使用原则如下：

1. 根层文档只保留六条核心要求和筛选法，不继续沉积场景条目。
2. 具体规则优先在 `framework/AGENTS.md`、workflow、skill、模板和维护文档中落地。
3. 同一条具体规则只保留一个主含义，不在多个高层文档里重复发明新表述。
4. 如果某条具体规则已不能清晰映射到六条核心要求，应优先降级、合并或删除。

下面是当前常见具体规则的归属映射：

| 具体规则 | 归属核心要求 | 说明 |
| --- | --- | --- |
| 目标市场适配 | 项目事实边界 | 这是项目面向谁、体验应服务谁的问题，不是新的核心要求 |
| 参考项目命名 / legacy 约束 | 项目事实边界 | 产出默认服务当前项目，而不是参考对象 |
| 共享基础能力先于业务模块 | 闭环与证据优先 | 本质上是先保证真实可运行闭环 |
| 目的与当前里程碑优先 | 闭环与证据优先 | 本质上是按当前交付目标判断优先级 |
| 一处定义 | 结构化演进 | 核心原则必须有主定义位置 |
| 变更需要证据 | 结构化演进 | 框架级调整不能只靠感觉 |
| 同步清理旧结构 | 结构化演进 | 新增或重构时必须处理历史残留 |

## 5. 当前阶段重点

现阶段 AI-OS 优先推进这些方向：

- 让第一次接触 AI-OS 的人更快理解“它解决什么问题、下一步怎么用”
- 让 workflow、模板和默认心智更少歧义，减少 AI 自动带偏
- 让关键工件和命令的命名更贴近用户能理解的语言
- 用测试和 eval 兜住反复出现的通用问题，而不是继续堆新的高层规则
