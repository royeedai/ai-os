## 1. 架构决策（Architecture Decisions）

### AD-001: 首版只做分诊，不做自动回复
- **决策**：第一阶段只交付分类、优先级和人工兜底，不接入自动回复能力
- **原因**：先稳定高风险链路，避免首版范围失控
- **备选方案**：同时上线自动回复，但会显著增加误判风险
- **影响范围**：`specs/ticket-triage.spec.md`、`tasks.yaml`、`acceptance.yaml`
- **来源**：需求澄清
- **状态**：生效中
- **日期**：2026-03-13

## 2. 编码约定（Conventions）

### CV-001: 分诊结果字段必须固定
- **约定**：所有分诊输出都必须包含 `category`、`priority`、`requires_human_review`
- **示例**：正确：完整结构化输出；错误：只返回分类不返回人工复核标记
- **来源**：spec
- **状态**：生效中
- **日期**：2026-03-13

## 3. 已知坑点（Pitfalls）

### PT-001: 不要只看分类正确率
- **问题**：只看最终分类准确率，会忽略高风险工单被错误自动流转
- **根因**：指标设计过于单一，没有覆盖人工兜底链路
- **绕行方案**：单独追踪高风险命中、人工拦截和错误自动流转
- **触发条件**：评估模型效果或上线前验收时
- **来源**：开发中发现
- **状态**：生效中
- **日期**：2026-03-13

## 4. 用户偏好（Preferences）

### PF-001: 示例项目优先展示工件闭环
- **偏好**：示例更强调 spec、tasks、acceptance、release、eval 的闭环，而不是业务代码细节
- **适用范围**：编写官方示例和 README 时
- **来源**：项目定位
- **状态**：生效中
- **日期**：2026-03-13

## 5. 硬性约束（Constraints）

### CT-001: 高风险工单必须人工复核
- **约束**：命中退款、封禁等敏感场景时必须将 `requires_human_review` 置为 `true`
- **原因**：涉及权限和用户资产风险，不能依赖自动流转
- **影响**：影响 spec、验收门禁、测试和 release 检查
- **来源**：业务约束
- **状态**：生效中
- **日期**：2026-03-13
