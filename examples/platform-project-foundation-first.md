# Example: Platform Project Foundation First

## 场景

你在做的平台型 / 后台型 / 流程型项目，通常会有这些特征：

- 多个业务模块共享登录、权限、导航、组织上下文、多语言、基础设置
- 单独完成一个业务模块页面，并不能形成真实可用闭环
- 第一阶段如果直接从“模块 A 页面”开始做，最后很容易看起来完成了，实际上跑不通

## 推荐做法

### 先用哪些 workflow

1. `/new-project`
2. 如有现有仓库或参考代码，再补 `/map-codebase`
3. 先完成基础能力层规划，再进入 `/new-module`

### 规划时要先明确

- 是否存在共享基础能力依赖
- 当前里程碑是不是“基础能力可用 + 首条核心业务闭环可运行”
- 哪些业务模块依赖这些基础能力，不能提前推进

### 第一阶段更合理的交付目标

不要写成：

- 完成入库模块
- 完成库存模块

更合理的是：

- 登录、权限、导航骨架可用
- 必要的组织 / 仓库 / 多语言 / 设置能力到位
- 一条核心业务链路能从入口跑到结果

## 典型工件

- `.ai-os/project-charter.md`
  - 明确共享基础能力依赖、默认交付等级、第一阶段里程碑
- `.ai-os/tasks.yaml`
  - 基础能力层优先于依赖它的业务模块
- `.ai-os/acceptance.yaml`
  - 第一阶段验收不是“某页面已做完”，而是“闭环已可运行”

## 不建议的做法

- 看到模块列表就按顺序做
- 因为用户先提了某个模块，就直接跳过基础能力层
- 把阶段目标写成“模块数量完成”
