# Example: Minimum Sufficient Change

## 场景

你要做的是局部改动，例如：

- 修一个纯 API 的字段映射问题
- 给现有命令补一个参数
- 调整一个工具类模块的输出格式
- 修一个不影响整体规划的小 bug

## 推荐做法

### 优先 workflow

1. `/quick`
2. 如果改动已经超出“小改动”边界，再切到 `/new-module`

### 判断标准

先问 3 件事：

1. 这是哪类模块：页面 / API / 数据处理 / 工具？
2. 这次改动属于 `L1`、`L2` 还是 `L3`？
3. 它是否真的需要新的 spec、完整发布流程或跨层联动检查？

### 合理的最小交付

- 保留任务记录和状态同步
- 保留最小需求说明
- 保留运行 / 构建 / 调用成功证据
- 只对当前模块相关的质量项做检查

## 不建议的做法

- 纯工具改动也默认补页面 / API / 数据库三件套说明
- 一个小 bug 修复也默认进入完整发布审批链
- 因为“更严格更安全”，就让所有改动都走最重路径

## 什么时候不该继续用 `/quick`

出现以下情况时，应升级为正式模块流程：

- 需求已经跨多个模块
- 需要新增长期维护的正式能力
- 需要补完整 spec、tasks、acceptance 闭环
- 存在明显的架构、权限、安全或发布风险
