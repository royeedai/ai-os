# 逆向分析器

本 Skill 用于把复刻项目的零散素材（截图、URL、API 入参返参、简短描述）转化为结构化的模块清单和 `.spec.md` 草稿，让 AI 从"写 spec"变成"审 spec"。

参考对象只用于帮助理解目标能力和界面，不自动意味着当前项目需要兼容、迁移或沿用源系统命名。

## 触发条件

- 用户说"复刻/仿制/抄一个系统"
- 用户提供了截图、URL、API 文档等素材要求还原
- `/clone-project` 工作流的阶段二自动调用

## 使用方式

### 第一步：素材收集与分类

1. 确认素材已放入项目 `.ai-os/references/clone-materials/` 目录
2. 使用 `references/material-checklist.md` 评估素材覆盖度
3. 按类型分类整理：

| 素材类型 | 存放位置 | 说明 |
|---------|---------|------|
| 截图 | `.ai-os/references/clone-materials/screenshots/` | 按模块或页面命名 |
| URL | `.ai-os/references/clone-materials/urls.md` | 列出所有可访问的 URL |
| API 文档 | `.ai-os/references/clone-materials/api/` | 接口入参返参、Swagger、Postman 等 |
| 文字描述 | `.ai-os/references/clone-materials/descriptions.md` | 需求简述、业务说明 |

### 第二步：逆向分析

对每种素材类型执行对应的提取规则：

#### 截图分析
- 提取页面结构：导航、侧边栏、主内容区、操作按钮
- 提取组件列表：表格列、表单字段、筛选条件、弹窗
- 提取交互暗示：按钮文案推断操作流程、Tab 推断子模块
- 提取导航关系：面包屑、菜单层级、Tab 切换
- 产出页面清单和功能需求草稿

#### URL 分析
- 提取路由结构和层级关系
- 推断模块划分和页面数量
- 如果 URL 可访问，提取页面标题、主要内容区域
- 产出模块清单和页面跳转关系

#### API 入参返参分析
- 提取字段名、字段类型、是否必填
- 推断数据模型和表结构
- 推断关联关系（如 `category_id` → 关联分类表）
- 推断状态枚举（如 `status: 0|1|2`）
- 推断业务规则（如价格字段 → 金额校验）
- 产出数据模型和 API 定义

#### 文字描述分析
- 提取业务上下文和核心场景
- 补充概述、目标用户、核心价值
- 产出验收标准草稿

### 第三步：产出 spec 草稿

1. 使用 `references/clone-spec-template.md` 为每个模块生成 `.spec.md` 草稿
2. 每条信息必须标记置信度：

| 标记 | 含义 | 示例 |
|------|------|------|
| `[确认]` | 从 API 文档、明确描述中直接获得 | 字段名、字段类型、接口路径 |
| `[推断]` | 从截图、命名规律、行业惯例推断 | 交互流程、状态机、权限模型 |
| `[待确认]` | 信息不足，需要用户或对方补充 | 业务规则、边界条件、非功能需求 |

3. 输出素材覆盖度评估：哪些模块有充足素材、哪些模块信息不足
4. 产出正文必须以**当前项目**为主体，参考对象只能放在"参考来源"章节中；除非用户明确要求，否则不要生成 `legacy`、"原项目"、"兼容旧系统"、源系统品牌名沿用等表述
5. 当参考对象中的命名会污染当前项目时，主动改写为当前业务语义，并把映射关系放入备注而不是正文命名

### 第四步：交付给用户审阅

1. 向用户展示：模块清单 + 每个模块的 spec 草稿 + 覆盖度评估
2. 用户逐项确认或修正 `[推断]` 和 `[待确认]` 条目
3. 用户确认当前项目命名、文案和模块边界是否已经脱离参考对象的 `legacy`/兼容叙事
4. 确认后的 spec 视为正式需求，进入 `/new-module` 开发流程

## 必须产出的内容

- 模块清单（含优先级和依赖关系）
- 每个模块的 `.spec.md` 草稿（使用 `clone-spec-template.md`）
- 素材覆盖度评估报告
- `[待确认]` 条目汇总清单

## 约束

- 不要在素材不足时自行脑补业务逻辑，必须标记 `[待确认]`
- 不要跳过素材分类直接开始分析，素材整理影响分析质量
- 不要遗漏 API 返参中的嵌套对象和数组字段
- 不要把截图中的装饰性元素当成功能需求
- 素材中发现的矛盾信息必须标记并提交用户决策
- 不要把参考对象当成兼容目标；除非用户明确要求，不得在输出中写入 `legacy`、"原项目"、"兼容旧系统" 之类措辞
- 不要沿用参考对象的品牌名、模块名或字段别名作为当前项目默认命名；应优先使用当前项目语义化命名

## 与其他 Skill 的关系

- **前置**：由 `/clone-project` 工作流调用
- **后续**：产出的 spec 交给 `spec-validator` 验证完整性
- **变更**：中途补充新素材时，通过 `change-impact-analyzer` 同步更新

## 模板引用

- 模块 spec 草稿模板：`references/clone-spec-template.md`
- 素材覆盖检查：`references/material-checklist.md`

### 示例：截图 + API 文档生成 spec 草稿

- 输入：页面截图、接口入参返参、简短业务描述
- 输出：模块清单、带 `[确认] / [推断] / [待确认]` 标记的 `.spec.md` 草稿
- 约束：无法从素材直接证明的信息必须保留为 `[待确认]`
