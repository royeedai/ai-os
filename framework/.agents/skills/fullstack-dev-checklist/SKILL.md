# 全栈开发质量检查清单

本 Skill 是 AI 开发全栈项目时的**强制检查清单**。每当开发一个页面、一个 API 接口或一个数据库模块时，必须逐项核对以下清单，确保没有遗漏。

## 适用边界

- 本清单是 V2 中的**默认全栈产品开发清单**，尤其适合后台、SaaS、管理系统、带前后端联动的业务模块。
- 如果项目是 CLI、SDK、数据管道、AI Agent App、纯 API 服务或轻量移动端，先调用 `project-planner` 选择 archetype，再按需组合本清单的相关维度。
- 不允许把本清单机械地套到所有项目类型上。

## 使用方式

1. 开发前：阅读对应模块的 `.spec.md` 文件，理解完整需求
2. 开发前：确认 `.ai-os-project/tasks.yaml` 和 `.ai-os-project/acceptance.yaml` 已定义当前模块的 DoR/DoD
3. 开发中：按本清单逐维度检查当前实现
4. 开发后：对照清单出具自检报告，标记每项的完成状态

## 维度零：工程完整性（编译与入口注册）🚧

**此维度为一切其他维度的大前提。未通过此维度的代码，后续所有审查都没有意义。**

### 编译/构建必须通过
- [ ] **编译验证**：是否执行了项目级编译/构建命令并确认无错误？
- [ ] **依赖完整**：依赖锁文件是否存在且与依赖声明文件保持同步？
- [ ] **字段类型合法**：Model 中的时间字段是否使用了正确的时间类型（而非其他类型）？金额字段是否标注了定点数 DB 类型？

### 变更感知验证必须匹配
- [ ] **验证基线存在**：是否存在 `.ai-os-project/verification-matrix.yaml`，并作为路径 -> 动作的单一真相源？
- [ ] **动作匹配**：本次改动是否根据 `verification-matrix.yaml` 推导出 `verification_required`、`restart_required`、`cold_start_required`？
- [ ] **运行时变更兜底**：若修改了配置、启动入口、DI、路由注册、共享代码、migration、worker 或构建配置，是否避免只依赖热更新验证？
- [ ] **重启证据**：若命中重启要求，是否留下 `restart-log`？
- [ ] **冷启动证据**：若命中冷启动要求，是否留下 `cold-start-log` 与 `post-restart-smoke-log`？

### 入口注册必须完整
- [ ] **路由注册**：新增的接口处理器是否已在路由配置文件中注册？
- [ ] **依赖注入**：新增的服务链路是否已在主入口文件中完成初始化和注入？
- [ ] **数据库迁移**：新增或修改的 Model 是否已加入数据库迁移脚本？
- [ ] **前端菜单/导航**：新页面是否已添加到侧边栏菜单/导航中，确保用户可以到达？

### 增量开发兼容
- [ ] **共享模型完整性**：当新模块引用已存在的 Model 时，旧 Model 是否已补全新模块所需的全部字段？
- [ ] **禁止跨文件重复声明**：同一目录下，是否存在两个文件定义了同名的结构体/类/函数？若有，合并到唯一的定义处。
- [ ] **关联入口同步**：更新了共享模型后，是否同步更新了迁移列表、Seed 脚本、测试 fixtures？

### 项目脚手架完整性
- [ ] **前端基础设施**：是否存在以下必需文件？
  - [ ] 依赖声明文件与启动脚本
  - [ ] 构建配置文件
  - [ ] HTML 入口文件
  - [ ] 应用初始化入口（UI 框架注册、插件加载）
  - [ ] 根组件（布局骨架、侧边栏导航、路由出口）
  - [ ] 路由表（每个页面组件都有对应路由）
  - [ ] 404 兜底页面（路由无匹配时展示友好提示）
  - [ ] 全局错误边界组件（防止单个组件报错导致整页白屏）
- [ ] **后端基础设施**：是否存在以下必需文件？
  - [ ] 依赖声明文件
  - [ ] 启动入口（数据库初始化 + 路由挂载 + 端口监听）
  - [ ] 路由注册文件 + 中间件配置（CORS、鉴权）
  - [ ] 文件上传处理接口（若业务涉及图片/文件上传）
- [ ] **通用基础设施**：是否存在？
  - [ ] 容器化构建文件
  - [ ] 本地环境编排文件（数据库 + 后端 + 前端）
  - [ ] 版本控制忽略规则文件
  - [ ] 环境变量示例文件（如 `.env.example`）及项目启动说明

### 本地可运行性
- [ ] **一键启动**：是否可通过一条命令启动完整的本地开发环境？
- [ ] **开发模式鉴权旁路**：后端是否支持开发模式环境变量跳过登录验证，便于前端开发调试？
- [ ] **前后端联调配置**：前端的 API 代理配置是否与后端实际监听地址匹配？
- [ ] **请求封装完备**：前端的 HTTP 工具封装是否已包含统一错误处理、Token 挂载、响应拦截？

## 维度一：页面完整性

### 列表页必须要素
- [ ] **CRUD 操作完整**：新增、查看详情、编辑、删除按钮是否齐全
- [ ] **批量操作**：是否需要批量删除、批量修改状态等
- [ ] **分页**：数据量大时是否有分页组件，分页参数是否正确传递
- [ ] **搜索/筛选**：是否提供关键字搜索和条件筛选
- [ ] **排序**：关键列（如创建时间、排序权重）是否可排序
- [ ] **状态筛选**：如有状态字段，是否有按状态筛选（如上架/下架、已支付/未支付）
- [ ] **操作列完整**：每行的操作按钮是否与业务逻辑匹配（编辑/删除/上下架/查看详情等）
- [ ] **表格列完整**：.spec 中定义的所有需要展示的字段是否都有对应列
- [ ] **筛选/下拉默认态**：所有下拉筛选器是否带有占位提示文字（如“请选择”“全部”）？禁止出现空白无提示的下拉框。
- [ ] **数据格式化 [P0]**：
  - 时间字段：是否全部格式化为人类可读格式（如 `YYYY-MM-DD HH:mm`）？**禁止直接展示原始 ISO 时间戳或 Unix 时间戳。**
  - 金额字段：是否带有单位（如 `¥`）并保留固定小数位？
  - 状态字段：是否用彩色标签/徽章展示，而非裸数字或普通文本？

### 表单页必须要素
- [ ] **字段完整**：.spec 中定义的所有字段是否都有对应表单项
- [ ] **字段类型合理**：
  - 关联数据（如分类）必须用**选择器/下拉框**并显示**名称**，绝不能让用户输入 ID
  - 日期用日期选择器，不用文本输入
  - 富文本内容用富文本编辑器
  - 图片用图片上传组件，并支持预览
  - 枚举字段用 Radio/Select，不用自由输入
- [ ] **下拉选择器占位符**：所有下拉框/选择器是否设置了“请选择”占位提示文字？
- [ ] **文件上传全链路 [P0]**：若业务涉及图片/文件上传，是否同时提供了：
  - 前端上传组件（带预览、删除重传）
  - 后端文件接收接口（保存到本地磁盘或云存储）
  - 返回可访问的 URL 供展示和存入数据库
- [ ] **必填标记**：必填字段有 `*` 标记
- [ ] **前端校验**：必填、格式（手机号/邮箱）、长度限制、数值范围
- [ ] **编辑回填**：编辑模式下所有字段是否正确回填原有数据
- [ ] **新增 vs 编辑**：新增和编辑是否共用表单组件，逻辑是否正确切换

### 详情页必须要素
- [ ] **信息完整**：所有关键字段是否展示
- [ ] **关联数据展示**：如主记录详情中展示关联子记录列表
- [ ] **操作按钮**：详情页是否有编辑、删除等跳转操作

## 维度二：API 完整性

### 每个接口必须检查
- [ ] **请求方法正确**：GET 用于查询，POST 用于创建，PUT/PATCH 用于更新，DELETE 用于删除
- [ ] **路径规范**：RESTful 风格，资源名用复数（`/api/products`）
- [ ] **参数校验**：
  - 必填参数是否校验（不能为空/零值）
  - 字符串长度限制
  - 数值范围限制
  - 枚举值是否在允许范围内
  - 关联 ID 是否存在性校验（外键引用是否有效）
- [ ] **分页支持**：列表接口是否支持 `page` + `page_size` 参数
- [ ] **分页边界保护**：page≥1、page_size 有上限（如100），超范围自动修正而非报错
- [ ] **排序支持**：列表接口是否支持 `sort_by` + `sort_order` 参数
- [ ] **筛选支持**：列表接口是否支持按条件筛选
- [ ] **权限校验**：接口是否检查用户登录状态和角色权限
- [ ] **错误响应规范**：
  - 统一的错误响应格式 `{ code, message, data }`
  - 区分 400（参数错误）、401（未登录）、403（无权限）、404（不存在）、500（服务器错误）
- [ ] **返回数据完整**：
  - 创建/更新后返回完整对象
  - 列表接口返回 `total` 总数
  - 关联数据是否预加载（如商品返回时包含分类名称）

### 接口文档
- [ ] 每个接口有清晰的请求/响应示例
- [ ] 字段类型和说明与 .spec 一致

### 日志与可观测性
- [ ] **操作日志**：关键操作（创建/更新/删除/状态变更）是否有结构化日志记录？
- [ ] **错误日志**：错误是否记录了请求上下文（用户 ID、请求参数、时间戳）？
- [ ] **后端路由兜底**：未知路由是否返回统一的 404 响应而非框架默认页面？

## 维度三：数据库设计

- [ ] **表名规范**：小写复数（`products`, `categories`, `orders`）
- [ ] **时间戳完整**：每张表都有 `created_at`, `updated_at`，软删除表有 `deleted_at`
- [ ] **索引合理**：
  - 外键字段有索引
  - 常用查询字段有索引（如 `status`, `category_id`）
  - 唯一约束字段有唯一索引
- [ ] **外键关系正确**：
  - 关联关系是否正确定义（一对多、多对多）
  - 级联操作策略是否合理（删除分类时商品怎么处理）
- [ ] **字段类型合理**：
  - 金额用定点数类型，不用浮点数
  - 状态用整数或字符串枚举
  - 长文本用 TEXT 类型
- [ ] **默认值**：状态字段等是否有合理默认值
- [ ] **软删除**：是否需要软删除（保留数据不物理删除）

## 维度四：用户体验（UX）

### 操作反馈
- [ ] **加载状态**：数据请求时显示 Loading 动画/骨架屏
- [ ] **空状态**：列表无数据时显示友好的空状态提示（不是空白页面）
- [ ] **成功反馈**：操作成功后有 Toast/消息提示
- [ ] **错误反馈**：请求失败时有明确的错误提示
- [ ] **确认弹窗**：删除等危险操作必须有二次确认弹窗，说明影响

### 导航与跳转
- [ ] **操作后跳转**：表单提交成功后自动跳转回列表页
- [ ] **返回按钮**：编辑/详情页有返回列表的入口
- [ ] **面包屑**：多层级页面有面包屑导航

### 表单体验
- [ ] **提交按钮状态**：提交中禁用按钮防重复提交
- [ ] **表单重置**：打开新增表单时清空上次填写内容
- [ ] **图片预览**：已上传图片可预览，可删除重传
- [ ] **实时校验**：关键字段失焦后即时校验，不要等到提交时

---

## 维度五：前后端一致性

- [ ] **字段名一致**：前端发送的字段名与后端接收的字段名完全匹配（驼峰 vs 下划线统一）
- [ ] **字段类型一致**：前端传 string 后端收 string，不存在类型不匹配
- [ ] **必填/可选一致**：前端标记必填的字段，后端也必须校验
- [ ] **枚举值一致**：前后端对状态值/类型值的定义相同
- [ ] **分页参数一致**：前后端分页字段名和逻辑一致（page 从 0 还是 1 开始）
- [ ] **错误处理一致**：前端能正确解析后端返回的所有错误格式

---

## 维度六：业务逻辑合理性

- [ ] **关联数据人性化展示**：
  - 关联实体显示名称而非裸 ID（后端预加载或 join 查询）
  - 创建人/操作人显示用户名而非 user_id
- [ ] **状态流转完整**：
  - 是否定义了状态机（每个状态的含义及允许的转换）
  - 每个状态下允许哪些操作，是否有严格的前置条件检查
- [ ] **级联影响处理**：
  - 删除/修改操作前是否检查了关联数据的约束条件
  - 状态变更前是否检查了未完成的关联业务
- [ ] **数据完整性**：
  - 涉及金额的操作是否在后端重新计算（不信任前端传入）
  - 快照数据是否在创建时锁定（不受后续修改影响）
- [ ] **权限逻辑**：
  - 普通用户只能操作自己的数据
  - 管理员可以操作所有数据
  - 不同角色的操作权限边界清晰

## 维度七：无障碍与核心体验 (A11y & Web Vitals) ♿

这是进阶的质量防线，确保产品"可用"且"好用"：
- [ ] **语义化 HTML**：是否正确使用了 `<nav>`, `<main>`, `<article>`, `<section>` 等语义标签，而非全是 `<div>`？
- [ ] **A11y (可访问性) 增强**：
  - [ ] 图标按钮或没有文本提示的控件，是否加上了 `aria-label` 或 `title`？
  - [ ] `<img>` 标签是否全部带有有意义的 `alt` 属性（或装饰性图片设为 `alt=""`）？
  - [ ] 键盘可依序导航吗？(`tabindex` 是否符合直觉逻辑，下拉框可否用 Enter/Space 选定)？
- [ ] **SEO 与 Meta**：若是面向 C 端页面，是否具备正确的 `<title>`, `<meta name="description">` 等 SEO 要素？
- [ ] **Web Vitals (性能建议)**：
  - [ ] 关键大图（如首屏 Banner、商品详情头图）是否限定了宽高等比例骨架，防 CLS 布局偏移？
  - [ ] 超大列表页是否做了虚拟滚动或分页规避 DOM 卡顿？
- [ ] **响应式适配**：页面是否在移动端也可用？关键布局是否做了响应式适配，或明确标记为“仅桌面端”？

## 维度八：双向溯源 (Spec-Code Traceability) 🔗

用于在代码验收与后期维护时，能顺藤摸瓜对齐产品需求。
- [ ] **需求注释锚点**：**[强制]** 开发核心逻辑（特别是复杂表单必填校验、关键状态机流转）时，是否能在代码边留下一句 `// 对应 .spec: FR-XXX`？
- [ ] **字段严格同质**：前后端的 DTO 变量名是否与 `.spec` 文档里`数据模型`部分定义的表头拼写**一字不差**，拒绝私自定义魔改缩写？
- [ ] **版本收敛**：业务发生变更时，是否是**先改 `.spec` 文件**再改代码？（严格禁止出现代码比文档更新的情况）。

---

## 联动指引

在完成本 Checklist 自查后，除了通用的测试，若涉及敏感业务还需引入专属检查：
1. 大中型重构/新架构评估 -> 调用 `architecture-reviewer`
2. 订单/支付接口上线前 -> 调用 `security-guard`
