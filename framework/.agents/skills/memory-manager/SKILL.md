# 记忆管理器

本 Skill 用于把"你已经提醒过很多次"的内容沉淀为结构化、可检索、可追溯的项目记忆。

## 使用方式

1. 使用下方模板生成或更新 `.ai-os/memory.md`
2. 每条记忆必须有**唯一 ID**、**分类**、**来源**、**状态**和**更新时间**
3. 只记录稳定事实、长期约束、明确决策、已证实的坑点
4. 临时讨论、易失状态、短期计划不要写进 memory
5. 若某条记忆已经失效，将状态标记为 `已过时`，附注原因，不要直接删除

## 适合写入 memory 的内容

- 架构决策（选型理由、方案对比、影响范围）
- 编码约定（命名规范、文件组织、API 风格）
- 已知坑点（踩过的坑、绕行方案、触发条件）
- 用户偏好（交互偏好、风格偏好、流程偏好）
- 硬性约束（技术限制、外部依赖限制、合规要求）

## 不适合写入 memory 的内容

- 今天临时要做的任务
- 尚未确认的猜测
- 随时会变化的运行状态
- 已记录在 `.ai-os/STATE.md` 中的当前进度

## 记忆条目规范

每条记忆必须包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| ID | 是 | 唯一标识，格式为 `{分类缩写}-NNN`（如 `AD-001`、`CV-003`） |
| 内容 | 是 | 记忆的具体内容 |
| 分类 | 是 | architecture / convention / pitfall / preference / constraint |
| 来源 | 是 | 产生这条记忆的事件（事故复盘 / 用户提醒 / 开发发现） |
| 状态 | 是 | 生效中 / 待确认 / 已过时 |
| 日期 | 是 | 创建或最后更新日期 |

## 模板

```markdown
# 项目记忆

## 元数据

- **最后更新**：[日期]
- **条目总数**：[数量]

## 1. 架构决策（Architecture Decisions）

### AD-001: [决策标题]
- **决策**：[具体选择]
- **原因**：[为什么这样选]
- **备选方案**：[考虑过但放弃的方案]
- **影响范围**：[影响哪些模块/文件]
- **来源**：[事故复盘 / 技术调研 / 用户要求]
- **状态**：生效中
- **日期**：[日期]

## 2. 编码约定（Conventions）

### CV-001: [约定标题]
- **约定**：[具体规则]
- **示例**：[正确 vs 错误写法]
- **来源**：[用户偏好 / 团队规范 / 框架要求]
- **状态**：生效中
- **日期**：[日期]

## 3. 已知坑点（Pitfalls）

### PT-001: [坑点标题]
- **问题**：[遇到了什么问题]
- **根因**：[为什么会发生]
- **绕行方案**：[如何避免]
- **触发条件**：[什么情况下会触发]
- **来源**：[开发中发现 / 事故复盘]
- **状态**：生效中
- **日期**：[日期]

## 4. 用户偏好（Preferences）

### PF-001: [偏好标题]
- **偏好**：[具体偏好]
- **适用范围**：[何时适用]
- **来源**：[用户明确表达 / 多次修正推断]
- **状态**：生效中
- **日期**：[日期]

## 5. 硬性约束（Constraints）

### CT-001: [约束标题]
- **约束**：[具体限制]
- **原因**：[为什么存在这个限制]
- **影响**：[对开发的具体影响]
- **来源**：[技术限制 / 合规要求 / 外部依赖]
- **状态**：生效中
- **日期**：[日期]
```
