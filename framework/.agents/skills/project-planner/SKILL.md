# 项目规划器

本 Skill 用于把模糊项目需求收敛成可交付的项目章程，避免 AI 在项目开始时就范围失控。

## 使用方式

1. 判断当前需求是否已经属于"项目级"而不是"单模块级"
2. 若项目存在用户可见界面，必须先确认目标市场 / 主要使用地区、主要用户习惯和体验参考方向；用户只说"企业项目"、"管理后台"、"ToB" 不算充分信息
3. 推荐至少追问以下内容：
   - 主要使用地区是中国大陆、海外还是多地区？
   - 更接近哪类产品习惯：国内业务后台 / 国内消费者产品 / 海外 SaaS / 内容产品 / 企业内网工具？
   - 是否有现成设计系统、品牌规范或明确参考产品？
4. 若用户回答模糊，只给出"企业用户"这类标签，必须继续追问，不得直接按 AI 默认风格产出
5. 读取 `references/archetypes.md`，选择最接近的项目 archetype
6. 根据 archetype 推荐交付等级（L1 探索 / L2 标准 / L3 高风险），用户可调整
7. 使用下方项目章程模板生成 `.ai-os/project-charter.md`
8. 使用下方风险清单模板生成 `.ai-os/risk-register.md`
9. 初始化 `.ai-os/STATE.md`（使用 `.agents/templates/project/STATE.md`）
10. 输出初版里程碑、模块清单、依赖和审批点
11. 对每个模块转交给 `spec-validator` / `task-orchestrator`
12. 若项目存在明显的共享基础能力依赖结构，先规划“基础能力层”与“首条业务闭环”，不要直接按模块列表顺序推进

## 必须产出的内容

- 项目目标与成功指标
- 范围内 / 范围外内容
- 目标用户和关键使用场景
- 目标市场 / 主要使用地区
- 体验 / 视觉参考方向（现有设计系统、品牌规范或默认交互风格）
- 非功能需求（性能、安全、稳定性、成本、适配范围 / 支持环境）
- 初版里程碑和模块拆分
- 基础能力层（如登录、权限、多语言、导航骨架、设置、数据字典、组织/仓库上下文）与业务模块的依赖关系
- 第一阶段的首条可运行业务闭环
- 外部依赖、风险、人工审批点
- 交付等级（L1 / L2 / L3）

## 约束

- 不要把"项目规划"写成大而空的愿景文档，必须能落到模块和里程碑
- 不要在范围边界不清时直接进入编码
- 不要默认所有项目都套 CRUD 模板，必须先选 archetype
- 对于复刻/参考驱动项目，参考对象只用于提取目标能力，不默认形成"兼容原系统"或"沿用 legacy 命名"要求；除非用户明确提出，否则项目章程和模块命名必须以当前项目为主体
- 对于用户可见界面，必须明确目标市场和体验参考方向；若面向中国大陆用户，默认使用更符合国内主流产品习惯的布局与交互，而不是 AI 默认的通用海外 SaaS 风格
- 若用户未明确目标市场，可给出结构化引导问题，但不得把"企业项目"自动等同于某一种设计习惯
- 对于存在共享基础能力依赖的项目，不要把“第一个业务模块”当作第一阶段目标；必须先规划基础能力层，并明确哪些设置项和平台能力是业务闭环的前置条件

## 交付输出

```text
.ai-os/project-charter.md
.ai-os/risk-register.md
.ai-os/STATE.md
模块清单
交付等级
下一步推荐的 workflow
```
