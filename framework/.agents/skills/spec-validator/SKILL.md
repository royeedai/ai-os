# .spec 文件完整性验证

本 Skill 用于确保 `.spec.md` 文件本身的质量。一个不完整的 spec 会导致开发时必然遗漏功能。

如果当前需求已经属于“新项目”或跨多个模块，先调用 `project-planner` 形成 `.ai-os/project-charter.md`，再为每个模块创建 `.spec.md`。

## 使用时机

- 创建新的 `.spec.md`
- 审阅或补齐现有 `.spec.md`
- 准备进入开发前，先确认 spec 是否足够支撑实现和验收

## 使用方式

1. 先识别当前 spec 的模块类型：`页面类` / `API 类` / `数据处理类` / `工具类`
2. 再识别当前模块的交付等级：`L1` / `L2` / `L3`
3. 使用本 Skill 检查“通用必填项 + 类型专项项 + 等级附加项”
4. 输出验证报告，标记缺失维度
5. 补齐缺失项后再交付给开发阶段

## 约束

- 如果当前需求还是项目级问题，不要直接写模块 spec，先回到 `project-planner`
- 不要为了“过模板”补空话；所有内容都应可执行、可验证
- 本 Skill 只验证 spec 本身，不替代编码、验收和发布检查

## 通用必填项

以下内容对所有模块类型都必须存在：

- [ ] 关联的 `.ai-os/project-charter.md`
- [ ] 模块类型
- [ ] 交付等级
- [ ] 所属里程碑 / 当前阶段
- [ ] 明确范围外内容
- [ ] 模块用途、目标用户、核心价值
- [ ] 功能需求（带唯一编号）
- [ ] 验收标准（可测试的条件）
- [ ] 交付证据要求（至少说明如何证明完成）

## 类型专项检查

### 页面类

- [ ] 界面 / 页面清单完整
- [ ] 页面跳转关系或入口路径明确
- [ ] 关键交互流程、字段组件类型、状态处理（加载/空/错/成功）明确
- [ ] 权限要求和边界条件明确
- [ ] 目标市场 / 主要使用地区、交互 / 视觉约束明确

### API 类

- [ ] 接口清单完整
- [ ] 请求参数、响应结构、错误处理明确
- [ ] 权限 / 鉴权 / 幂等或并发处理明确
- [ ] 数据模型或持久化影响明确（如涉及）
- [ ] 契约验证方式明确（接口样例、契约测试或等价证据）

### 数据处理类

- [ ] 输入源、触发方式（手动 / 定时 / 事件）明确
- [ ] 处理逻辑、容错 / 重试 / 补数策略明确
- [ ] 输出位置、结果格式、结果校验方式明确
- [ ] 日志、告警、失败处理或人工介入方式明确
- [ ] 幂等、重复执行、回放策略明确（如涉及）

### 工具类

- [ ] 命令 / 子命令 / 入口清单明确
- [ ] 参数、配置、默认值、输出格式明确
- [ ] 安装 / 运行方式、依赖环境、失败退出方式明确
- [ ] 使用示例或关键命令样例明确
- [ ] 若涉及兼容性或版本策略，说明影响范围和约束

## 等级附加项

### L1 探索

- [ ] 允许轻量 spec，但必须保留模块目标、范围边界、核心路径和最小验证方式
- [ ] 若省略完整章节，仍应说明为什么当前阶段可以简化

### L2 标准

- [ ] 类型专项项完整
- [ ] 验收标准覆盖正常流程和关键异常流程
- [ ] 交付证据足以支撑 `acceptance-gate`

### L3 高风险

- [ ] L2 全部满足
- [ ] 风险、审批点、回滚 / 发布影响明确
- [ ] 安全、权限、数据风险或外部依赖风险明确
- [ ] 需要额外的安全 / 架构 / 发布检查时已写入 spec 或关联工件

## 派生规则检查

按 `.agents/references/derived-rules.md` 额外检查：

- [ ] 目标市场适配是否已写成可执行信息，而不是“企业项目”“管理后台”“ToB”这类模糊标签
- [ ] 若是复刻 / 参考驱动项目，参考对象是否仅保留在“参考来源”或等价章节，而未把 `legacy`、`原项目`、`兼容旧系统`、源系统品牌名扩散到正文命名、验收项和用户文案
- [ ] 若当前模块受共享基础能力依赖约束，spec 是否明确前置依赖，而不是假定这些能力已存在

## 常见遗漏信号

- [ ] 为了“过模板”而补空话，但没有可执行判断标准
- [ ] 把所有模块都写成页面 + API + 数据库三件套，即使实际并不需要
- [ ] 没有写清本模块不做什么
- [ ] 只写功能描述，没有写验证方式
- [ ] 只写 happy path，没有写关键异常或边界条件

## 验证报告模板

```markdown
## .spec 验证报告：[模块名]

- 模块类型：[页面类 / API 类 / 数据处理类 / 工具类]
- 交付等级：[L1 / L2 / L3]
- 完整性结论：[通过 / 需补充]

### 缺失项
1. [缺失内容]

### 风险提示
1. [若继续推进会导致的风险]

### 建议补充
1. [应补的具体信息]
```
