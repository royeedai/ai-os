# .spec 文件完整性验证

本 Skill 用于确保 `.spec.md` 文件本身的质量。一个不完整的 .spec 会导致开发时必然遗漏功能。

如果当前需求已经属于“新项目”或跨多个模块，先调用 `project-planner` 形成 `.ai-os/project-charter.md`，再为每个模块创建 `.spec.md`。

## 使用方式

1. 编写 .spec 文件后，使用本 Skill 逐项检查 .spec 的完整性
2. 输出验证报告，标记缺失的维度
3. 补充缺失项后再交付给开发阶段

## .spec 文件必须包含的章节

### 0. 项目关联（必须）
- [ ] 关联的 `.ai-os/project-charter.md` 或上级项目名称
- [ ] 当前模块所属里程碑 / 任务阶段
- [ ] 明确本模块**不包含**的范围边界

### 1. 概述（必须）
- [ ] 模块用途描述
- [ ] 目标用户（谁来用这个功能）
- [ ] 目标市场 / 主要使用地区（尤其是用户可见界面）
- [ ] 核心价值（为什么需要这个模块）

### 2. 页面/接口清单（必须）
- [ ] 列出本模块的**所有**页面或接口
- [ ] 每个页面/接口有简短描述
- [ ] 页面之间的跳转关系明确

### 3. 功能需求（必须）
每个功能需求必须包含：
- [ ] **唯一编号**（如 FR-001）
- [ ] **功能描述**：具体做什么
- [ ] **交互流程**：用户怎么操作（步骤 1→2→3）
- [ ] **数据字段**：涉及哪些字段，每个字段的类型、是否必填、校验规则
- [ ] **状态处理**：加载中 / 空数据 / 错误 / 成功 各怎么展示
- [ ] **权限要求**：谁能看、谁能操作
- [ ] **边界条件**：极端情况怎么处理（大数据量、并发、空值等）

### 4. 数据模型（必须）
- [ ] 涉及的数据库表和核心字段
- [ ] 字段类型和约束
- [ ] 关联关系

### 5. API 定义（必须，如涉及后端）
- [ ] 接口 URL 和方法
- [ ] 请求参数（名称、类型、必填、说明）
- [ ] 响应结构（成功和失败）
- [ ] 错误码定义

### 6. 非功能需求（推荐）
- [ ] 性能要求
- [ ] 安全要求
- [ ] 适配范围 / 支持环境要求
- [ ] 交互 / 视觉约束（如设计系统、目标市场习惯、信息密度要求）

### 7. 关联模块（推荐）
- [ ] 依赖哪些其他模块
- [ ] 影响哪些其他模块

### 8. 验收标准（必须）
- [ ] 可测试的具体条件（Given-When-Then）
- [ ] 至少覆盖正常流程和 2-3 个异常流程

### 9. 交付证据（推荐）
- [ ] 如何证明该模块已完成（构建、测试、截图、日志、样例请求）
- [ ] 是否要求迁移、回滚或上线后的 Smoke Check

## 常见遗漏维度检查

以下是 .spec 中**最容易遗漏**的内容，必须特别检查：

### 列表页常见遗漏
- [ ] 是否定义了每列显示什么数据、什么格式
- [ ] 是否定义了操作列有哪些按钮
- [ ] 是否定义了搜索/筛选条件有哪些
- [ ] 是否定义了默认排序规则
- [ ] 是否定义了空数据时的展示

### 表单页常见遗漏
- [ ] 是否定义了每个字段的**输入组件类型**（而非只写字段名）
- [ ] 是否定义了关联数据的选择来源（如分类下拉框的数据从哪个接口获取）
- [ ] 是否定义了图片上传的格式/大小限制
- [ ] 是否定义了新增和编辑的差异点

### 业务逻辑常见遗漏
- [ ] 是否定义了状态机（每个状态的含义及允许的转换）
- [ ] 是否定义了删除操作的前置检查（关联数据如何处理）
- [ ] 是否定义了数据的所有权（谁能看到哪些数据）
- [ ] 是否定义了排序权重的默认值和调整方式

### 展示规格常见遗漏
- [ ] 是否定义了时间字段的展示格式（如 `YYYY-MM-DD HH:mm`）
- [ ] 是否定义了金额字段的展示格式（单位、小数位数）
- [ ] 是否定义了状态字段的展示样式（标签颜色映射）
- [ ] 是否定义了下拉/筛选器的默认状态（占位提示文字）
- [ ] 是否定义了文件上传的完整约束（格式/大小/存储方式/上传接口）

### 跨领域考量常见遗漏
- [ ] 是否明确了是否需要多语言支持？（若需要，文本是否走 i18n 配置）
- [ ] 是否明确了目标设备和支持环境？（仅桌面端 / 移动端 / 响应式 / 浏览器范围 / 运行环境）
- [ ] 是否明确了目标市场 / 主要使用地区，以及界面是否需要贴合国内主流产品习惯、海外产品习惯或现有设计系统？
- [ ] 若项目只有"企业项目"、"管理后台"、"ToB" 这类表述，是否继续追问并补充到可执行的市场与体验信息，而不是直接接受模糊描述？
- [ ] 对于复刻/参考驱动项目，是否已将参考对象限制在"参考来源"章节，而没有把 `legacy`、"原项目"、"兼容旧系统"、源系统品牌名扩散到正文命名、验收项和用户文案？
- [ ] spec 变更时，是否标记了变更的 FR 编号并通知对应代码的修改需求？
- [ ] 是否明确了该模块需要哪些完成证据，供 `acceptance-gate` 使用？

## 验证报告模板

```markdown
## .spec 验证报告：[模块名]

### 完整性评分：X/10

| 章节 | 状态 | 缺失项 |
|------|------|--------|
| 项目关联 | ✅ | - |
| 概述 | ✅ | - |
| 页面清单 | ✅ | - |
| 功能需求 | ⚠️ | 缺少编辑页的字段校验规则 |
| 数据模型 | ✅ | - |
| API 定义 | ❌ | 缺少错误码定义 |
| 验收标准 | ⚠️ | 缺少异常流程测试条件 |
| 交付证据 | ⚠️ | 缺少截图和测试结果要求 |

### 建议补充
1. [具体描述需要补充的内容]
```

请参阅 [spec-template.md](references/spec-template.md) 获取完整的 .spec 文件模板。
