# 新项目启动流程

当用户说"做一个新项目"、"从 0 开始搭一个系统"、"规划一个完整产品"时触发此流程。

## 阶段一：项目定义

1. 先确认项目是否存在用户可见界面；若存在，必须追问目标市场 / 主要使用地区、主要用户习惯和体验参考方向
2. 当用户只说"企业项目"、"管理后台"、"ToB" 时，继续追问，而不是直接假设设计风格：
   - 主要用户在中国大陆、海外，还是跨地区？
   - 更像国内业务后台，还是海外 SaaS 工具？
   - 有没有现成设计系统、品牌规范或参考产品？
3. 调用 `project-planner`，选择项目 archetype
4. 根据 archetype 推荐交付等级（L1 探索 / L2 标准 / L3 高风险），用户可调整
5. 创建 `.ai-os/project-charter.md`，明确目标、范围边界、目标用户、目标市场、体验参考方向、成功指标、非功能需求、依赖、审批点、交付等级
6. 若项目存在共享基础能力依赖结构，先规划基础能力层：登录、权限、多语言、导航骨架、组织/工作区/仓库上下文、基础设置、数据字典
7. 为这类项目优先定义第一阶段目标为“基础能力可用 + 首条核心业务闭环可运行”，而不是只选一个业务模块先做
8. 识别项目不包含的内容，避免 AI 自动脑补范围
9. 产出初版里程碑和模块清单

## 阶段二：任务编排

10. 调用 `task-orchestrator`，为里程碑和模块生成 `.ai-os/tasks.yaml`（含 wave 并行分组和 context_files）
11. 初始化 `.ai-os/verification-matrix.yaml`，声明关键组件、build/start/restart 命令和冷启动 Smoke 基线
12. 为关键模块预留 `.ai-os/specs/*.spec.md`、测试、发布、回滚、文档任务，不允许只拆开发任务
13. 调用 `acceptance-gate` 生成项目级 `.ai-os/acceptance.yaml`
14. 建立 `.ai-os/risk-register.md`，标出外部依赖、风险等级和人工审批点
15. 初始化 `.ai-os/STATE.md`，记录项目当前位置和进度

## 阶段三：确认与启动

16. 向用户展示项目章程、模块清单、关键任务图、验证基线、交付等级、主要风险
17. 对存在共享基础能力依赖的项目，显式向用户回显基础能力层、首条业务闭环和被延后的后续模块，确认阶段目标无误
18. 明确向用户回显目标市场、体验参考方向和默认假设，确认无误后再进入实现阶段
19. 用户确认后，再进入具体模块的 `/new-module`
20. 若在已有代码库上开发，建议先执行 `/map-codebase`
21. 若后续范围变化，转入 `/change-request`
