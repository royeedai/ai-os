# 新项目启动流程

当用户说"做一个新项目"、"从 0 开始搭一个系统"、"规划一个完整产品"时触发此流程。

## 阶段一：项目定义

1. 调用 `project-planner`，选择项目 archetype
2. 若项目存在用户可见界面，按 `.agents/references/derived-rules.md` 的“目标市场适配”规则确认目标市场、主要使用地区和体验参考方向
3. 根据 archetype 推荐项目默认交付等级，并为每个核心模块补充模块类型（`页面类` / `API 类` / `数据处理类` / `工具类`）和默认交付等级
4. 创建 `.ai-os/project-charter.md`，明确目标、范围边界、目标用户、目标市场、体验参考方向、成功指标、非功能需求、依赖、审批点、模块类型和交付等级
5. 若项目存在共享基础能力依赖结构，按 `.agents/references/derived-rules.md` 的“共享基础能力优先”规则先规划基础能力层
6. 为这类项目优先定义第一阶段目标为“基础能力可用 + 首条核心业务闭环可运行”，而不是只选一个业务模块先做
7. 识别项目不包含的内容，避免 AI 自动脑补范围
8. 产出初版里程碑和模块清单

## 阶段二：任务编排

9. 调用 `task-orchestrator`，为里程碑和模块生成 `.ai-os/tasks.yaml`（含 wave 并行分组和 context_files）
10. 初始化 `.ai-os/verification-matrix.yaml`，声明关键组件、build/start/restart 命令和冷启动 Smoke 基线
11. 为关键模块预留 `.ai-os/specs/*.spec.md`、测试、发布、回滚、文档任务，不允许只拆开发任务
12. 调用 `acceptance-gate` 生成项目级 `.ai-os/acceptance.yaml`
13. 建立 `.ai-os/risk-register.md`，标出外部依赖、风险等级和人工审批点
14. 初始化 `.ai-os/STATE.md`，记录项目当前位置和进度

## 阶段三：确认与启动

15. 向用户展示项目章程、模块清单、模块类型、交付等级、关键任务图、验证基线和主要风险
16. 对存在共享基础能力依赖的项目，显式向用户回显基础能力层、首条业务闭环和被延后的后续模块，确认阶段目标无误
17. 明确向用户回显目标市场、体验参考方向和默认假设，确认无误后再进入实现阶段
18. 用户确认后，再进入具体模块的 `/new-module`
19. 若在已有代码库上开发，建议先执行 `/map-codebase`
20. 若后续范围变化，转入 `/change-request`
