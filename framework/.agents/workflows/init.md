# 老项目初始化流程

当用户说"初始化项目"、"初始化老项目"、"给项目建基础文件"、"init"时触发此流程。适用于已有代码库第一次接入 AI-OS，需要一次性生成所有基础项目文件。

## 何时使用

- 已有代码库第一次接入 AI-OS（已运行 `npx create-ai-os .` 安装框架文件）
- 项目已经在开发中，需要补齐 AI-OS 项目管理文件
- 想用 AI-OS 管理一个成熟项目的后续开发

## 不该使用

- 从零开始的新项目 → 用 `/new-project`
- 只需要分析代码库，不需要生成项目文件 → 用 `/map-codebase`
- 只是做一个小改动 → 用 `/quick`

## 会生成什么

所有文件均基于对现有代码库的分析填入**真实内容**，而非空模板：

- `.ai-os/project-charter.md` — 项目章程（目标、范围、技术栈、里程碑、模块拆分）
- `.ai-os/codebase-map.md` — 代码库分析报告（技术栈、架构、约定、现有资产）
- `.ai-os/tasks.yaml` — 任务图（基于项目当前状态）
- `.ai-os/STATE.md` — 项目状态仪表盘
- `.ai-os/memory.md` — 项目记忆（稳定约束、编码约定）
- `.ai-os/verification-matrix.yaml` — 验证矩阵（构建/启动/重启命令）
- `.ai-os/risk-register.md` — 风险登记

## 步骤

### 阶段一：代码库分析

1. **技术栈识别**：扫描项目结构，识别语言、框架、包管理器、构建工具、测试框架、数据库
2. **架构分析**：分析代码组织模式（分层结构、模块边界、目录职责）
3. **约定提取**：识别命名约定（变量、文件、API 路径、数据库表）
4. **模式识别**：提取现有模式（错误处理、鉴权、数据访问、API 风格、状态管理、环境配置）
5. **运行拓扑识别**：梳理现有服务/进程、build/start/restart 命令、冷启动入口
6. **现有资产盘点**：识别已有的数据库 schema、API 端点、页面路由、共享组件
7. **目标市场与体验风格推断**：若项目存在用户可见界面，按 `.agents/references/derived-rules.md` 的“目标市场适配”规则，从语言、币种、文案、日期格式、导航结构、现有设计系统、已上线域名等信号推断主要使用地区和体验风格；若无法确定，明确标记为“待确认”
8. **生成 `.ai-os/codebase-map.md`**：将以上分析结果写入代码库分析报告

### 阶段二：项目文件初始化

9. 调用 `project-planner`，根据代码库分析结果选择项目 archetype，并为核心模块补齐模块类型和默认交付等级
10. **创建 `.ai-os/project-charter.md`**：基于分析结果填入真实内容：
   - 项目目标：从 README、package.json 等提取
   - 范围边界：基于现有模块和功能
   - 技术栈和非功能需求：基于代码库分析
   - 目标市场和体验参考方向：基于现有界面与产品信号推断，并标记已确认 / 待确认
   - 里程碑：基于现有进度
   - 模块拆分：基于目录结构和架构分析，并补充模块类型、默认交付等级、是否受共享基础能力依赖约束
11. **初始化 `.ai-os/verification-matrix.yaml`**：声明构建/启动/重启命令和冷启动 Smoke 基线
12. **初始化 `.ai-os/tasks.yaml`**：创建初始任务图，可以是空的 wave 结构或基于已知待办
13. **初始化 `.ai-os/STATE.md`**：记录项目当前位置（已有代码库，初始化完成阶段）
14. **初始化 `.ai-os/memory.md`**：将分析中发现的稳定约束和编码约定写入项目记忆
15. **初始化 `.ai-os/risk-register.md`**：基于项目特征记录已知风险

### 阶段三：确认与启动

16. 向用户展示所有生成的文件清单和关键内容摘要
17. 若目标市场 / 体验风格是推断而非确认，必须显式告知用户并请求确认，不得把推断当成既定事实
18. 用户确认后，建议下一步：
    - 需要加新功能 → `/new-module`
    - 需要修小 bug → `/quick`
    - 需求变化 → `/change-request`
19. 提醒用户将 `.ai-os/` 目录提交到版本控制

## 与其他 workflow 的关系

- `/init` 整合了 `/map-codebase`（代码分析）和 `/new-project`（项目文件创建）的关键步骤
- `/init` 完成后，后续开发直接使用 `/new-module`、`/quick` 等 workflow
- 如果项目需要更深入的代码分析，可以再运行 `/map-codebase` 补充
