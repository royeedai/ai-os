# 派生规则参考

本文件收口来自 AI-OS 核心要求的常见派生规则，供 `framework/AGENTS.md`、workflow、skill、模板引用。

这些规则不是新的高层原则，而是对现有核心要求的具体落地。

## 1. 项目事实边界

### 1.1 目标市场适配

- 对于存在用户可见界面的项目，必须确认目标市场 / 主要使用地区、体验参考方向，以及是否存在既有设计系统或品牌规范
- 当用户只说“企业项目”“管理后台”“ToB”这类宽泛标签时，信息仍然不足，必须继续追问
- 若项目主要面向中国大陆用户，默认采用更贴近国内主流产品习惯的信息密度、导航结构、表格与筛选布局、表单组织方式和文案语气
- 若项目已有设计系统、品牌规范，或目标市场不是中国大陆，则以既有体系和实际目标市场为准
- 若目标市场只能推断，必须把推断写进工件并标记为“待确认”，不得静默按 AI 默认风格产出

### 1.2 参考来源命名边界

- 参考对象只是信息来源，不是默认兼容目标
- 产出默认以当前项目为主体命名和描述
- 除非用户明确要求，否则不要自动引入 `legacy`、`原项目`、`兼容旧系统`、源系统品牌名等表述
- 差异、替换和取舍应被显式写出，而不是把参考对象的包袱带入正文

## 2. 闭环与证据优先

### 2.1 当前里程碑优先

- 接到 Bug、想法、优化或需求变更时，先读取 `.ai-os/project-charter.md` 和 `.ai-os/STATE.md`
- 判断事项是否直接阻塞当前项目目标、当前里程碑目标、当前核心验收路径或当前进行中的任务
- 若不阻塞当前目标，且用户未明确调整优先级，默认进入待排期，而不是直接打断当前主线

### 2.2 共享基础能力优先

- 若多个业务模块共同依赖登录、权限、多语言、导航骨架、组织/工作区/仓库上下文、基础设置、数据字典等基础能力，必须先规划这些依赖
- 第一阶段目标应优先达到“基础能力可用 + 首条核心业务闭环可运行”，而不是只完成孤立模块页面
- 在依赖未具备时，不得把业务模块标记为正式进入实现；应先补齐前置能力或重排里程碑
- 阶段验收不得接受“孤立模块完成但整体不可运行”的伪完成状态
